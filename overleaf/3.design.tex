\chapter{Project Design}
\label{ch:project_design}

This chapter presents the complete architectural blueprint, requirement analysis, behavioral feature engineering specification, and mathematical formulation framework of the proposed passive anxiety detection system. The design described herein reflects the actual system that was built and deployed during the study: a zero-intrusion, OS-level behavioral monitoring tool for first-year undergraduate students working in the Code::Blocks C/C++ integrated development environment during graded academic examination sessions. No wearable sensors, IDE plugins, or artificial experimental environments were used at any stage.

% ─────────────────────────────────────────────
\section{Requirement Analysis}
\label{sec:requirements}

\subsection{Functional Requirements}
The functional requirements define the exact behavioral boundaries of the implemented system:

\begin{itemize}
    \item The system shall run a non-intrusive background monitoring process that passively captures keystroke timing metadata and active window context transitions without interfering with the student's primary Code::Blocks programming environment.
    \item The system shall detect and log the state of the compiler executable (\texttt{gcc.exe}) via the Windows \texttt{tasklist} command to track compilation invocations and success rates during each 30-second observation interval.
    \item The system shall aggregate 18 behavioral features per interval and compute a rule-based \texttt{anxiety\_score} (range: 0--100) and a categorical \texttt{risk\_level} (\texttt{LOW}, \texttt{MODERATE}, \texttt{HIGH}, \texttt{CRITICAL}) at the end of each 30-second tumbling window.
    \item The system shall serialize all computed behavioral feature rows into a local Comma-Separated Values (CSV) log file, strictly excluding any literal keystroke characters, alphanumeric input sequences, or source code content from the stored records.
    \item The system shall be silently deployable via a batch (\texttt{.bat}) script wrapper, enabling the researcher to initialize the monitor on each student workstation at the start of an examination session without student awareness.
    \item The system shall produce a structured, machine-learning-ready dataset suitable for post-session offline statistical validation and supervised binary classification experiments.
\end{itemize}

\subsection{Non-Functional Requirements}
\begin{itemize}
    \item \textbf{Performance:} The background monitoring process has to run on its own at the operating system level. It should not use up a lot of the computer's processing power. The 30-second cycle to collect data should not slow down the Code::Blocks compilation pipeline.
    \item \textbf{Privacy:} The system should only record what people do. How they use the system. It must never record what people type, like passwords or parts of their code. It can only save things like how people take to do something, how many times they do it and what kind of windows they have open.
    \item \textbf{Scalability:} The \texttt{.bat} We should be able to set up the launcher and monitoring script on all the computers in a university lab at the time without needing any special network equipment or internet services. We also should not need to install any plugins for the coding software.
    \item \textbf{Reproducibility:} The data we collect should be organized in a way that anyone else can use it to get the results. They should be able to start from scratch, clean the data look at it and use it to make predictions and they should get the results that we did.
    \item \textbf{Portability:} The system should only use the Python library and the normal Windows functions so we do not need to rely on special software or licenses that cost money.
    \item \textbf{Security:} All the data we collect should be completely anonymous. We identify each student's session with a code on their own computer rather than with their name or any other personal information. This way we follow all the rules for handling data in a way that's fair and respectful to the students.
\end{itemize}

\section{System Architecture}
\label{sec:system_architecture}

\subsection{Overview}
The system is architecturally decoupled into two fully independent execution phases: the \textbf{Live Data Collection Phase}, which runs silently on student machines throughout the duration of each examination, and the \textbf{Offline Research Analysis Phase}, which is executed by the researcher after all exam sessions have concluded. This strict separation guarantees zero runtime interference with the student's primary coding environment and allows the full machine learning pipeline to be re-executed independently of the live monitoring system at any future point.

Figure~\ref{fig:system-architecture} illustrates the complete runtime architecture of the passive anxiety monitor, showing the three parallel sensing threads and the central 30-second aggregator engine.

\begin{figure}[htbp]
    \centering
    \includegraphics[width=0.95\textwidth]{00_system_architecture.png}
    \caption{Runtime Architecture of the Passive Anxiety Monitor: Three OS-Level Sensing Threads Converging into the 30-Second Behavioral Aggregator.}
    \label{fig:system-architecture}
\end{figure}

\subsection{Architectural Layer Breakdown}
The live collection pipeline consists of three concurrent threads working with sensing and merging into one aggregation engine every 30 seconds: 

\begin{enumerate}
    \item \textbf{Keyboard Hook Thread (100 Hz):}  The thread hooks into a low-level keyboard hook with the Windows OS event queue using the \texttt{SetWindowsHookEx} WinAPI function. In this way, it is able to capture the precise high-resolution timestamp of every key-press and key-release event without registering the character value of any key. Using raw timestamps series, it calculates the inter-keystroke interval (IKI), mean typing velocity in words per minute (WPM), total keystroke number, backspaces number, typing pauses, and idle windows.

    \item \textbf{Window Focus Tracker:} The tracker regularly checks the \texttt{GetForegroundWindow()} and \texttt{GetWindowText()} WinAPI functions to figure out the current foreground-focused process. It classifies the window into one of the following three semantic categories: Code::Blocks editor (\texttt{codeblocks.exe}), terminal or runtime console window (\texttt{cmd.exe}), or external application (for example, a PDF document reader that is used for reading the problem description). Every switching between different categories is considered as a \texttt{focus\_switch} event; the total time spent on each category during every 30-second window is counted.

    \item \textbf{Compile Event Detector:} The detector regularly invokes the \texttt{tasklist} Windows command-line utility for checking the presence of \texttt{gcc.exe} in the active process list. Every such detection is marked as a compilation invocation. Moreover, the detector records the success status of each compilation attempt (\texttt{compile\_success\_rate}) and attempts themselves (\texttt{compile\_attempts}).
\end{enumerate}

\noindent Every 30 seconds at the end of tumbling window, the \textbf{30-Second Aggregation Engine} aggregates all raw logs from the above-described three threads, calculates the full 18-feature behavior vector, applies the rule-based anxiety scoring formula, and writes the resulted structured row into the student's local CSV log file. The Code::Blocks IDE and the student's work session are not affected by the whole process.

\section[Mathematical Formulation of Features]{Mathematical Formulation of Behavioral Feature Telemetry}
\label{sec:math}

To translate noisy, unconditioned human-computer interaction logs into a structured high-frequency feature space, the system applies a set of formal mathematical definitions to each 30-second observation interval $\Delta t$. Let $K$ represent the total number of recorded key-release events within an interval, and let $T_i$ represent the precise high-resolution timestamp of the $i$-th key event.

\subsection{Mean Typing Velocity ($V_{\text{WPM}}$)}
The mean typing velocity measures the rate of keyboard output normalized to a standard words-per-minute metric, assuming an average word length of five characters:
\begin{equation}
V_{\text{WPM}} = \frac{\left(\frac{K}{5}\right)}{\Delta t \times \frac{1}{60}} = \frac{12 \times K}{\Delta t}
\end{equation}
A sudden drop or spike in $V_{\text{WPM}}$ across consecutive windows serves as a reliable indicator of abrupt changes in cognitive engagement or task difficulty.

\subsection{Inter-Keystroke Latency Variance ($\sigma^2_{\text{Lat}}$)}
Let the individual latency interval $L_i$ define the precise duration between two successive key-release timestamps, such that $L_i = T_i - T_{i-1}$. The mean inter-keystroke latency over the full window interval is:
\begin{equation}
\mu_{\text{Lat}} = \frac{1}{K-1} \sum_{i=2}^{K} L_i
\end{equation}
The typing latency variance isolates erratic, non-rhythmic typing patterns that serve as the strongest measurable signature of underlying cognitive hesitation or situational panic:
\begin{equation}
\sigma^2_{\text{Lat}} = \frac{1}{K-2} \sum_{i=2}^{K} \left(L_i - \mu_{\text{Lat}}\right)^2
\end{equation}
To eliminate mechanical outliers caused by external disruptions (such as a student pausing to ask the invigilator a question), the computed variance is bounded by a hard upper-capping threshold during preprocessing:
\begin{equation}
\sigma^2_{\text{Lat,\,capped}} = \min\!\left(\sigma^2_{\text{Lat}},\; 30{,}000\;\text{ms}\right)
\end{equation}

\subsection{Backspace Rate Index ($R_{\text{BS}}$)}
Let $B$ represent the total count of backspace or delete key-press events recorded within the interval. The backspace rate is expressed as a percentage of all recorded input activity:
\begin{equation}
R_{\text{BS}} = \left(\frac{B}{K + B}\right) \times 100
\end{equation}
Elevated values of $R_{\text{BS}}$ indicate repetitive error-correction behavior, which HCI research associates strongly with programmer frustration and elevated cognitive load.

\subsection{Spatial Environment Fragmentation Index ($F_{\text{Switch}}$)}
Let $W_t$ represent the classified window category occupying the OS foreground focus at micro-timestamp $t$ (one of: \texttt{codeblocks}, \texttt{terminal}, \texttt{other}). The context-switching index counts all transitions between distinct window categories within the interval:
\begin{equation}
F_{\text{Switch}} = \sum_{t} \mathbb{I}\!\left(W_t \neq W_{t-\delta}\right)
\end{equation}
where $\mathbb{I}(\cdot)$ is the indicator function, returning 1 when the condition is true and 0 otherwise. A rapid increase in $F_{\text{Switch}}$ captures the phenomenon of Session Fragmentation, providing a robust and entirely non-intrusive signature of severely elevated mental workload.

\subsection{Rule-Based Anxiety Score ($S_{\text{Anxiety}}$)}
Before the deployment of supervised machine learning classifiers, a heuristic rule-based engine was constructed to produce a continuous scalar anxiety estimate from the behavioral feature vector. The engine applies a weighted linear combination of the primary distress indicators:
\begin{equation}
S_{\text{Anxiety}} = \alpha_1 \cdot \hat{\sigma}^2_{\text{Lat}} + \alpha_2 \cdot R_{\text{BS}} + \alpha_3 \cdot \hat{P}_{\text{ratio}} + \alpha_4 \cdot F_{\text{Switch}} + \alpha_5 \cdot (1 - C_{\text{success}})
\end{equation}
where $\hat{\sigma}^2_{\text{Lat}}$ and $\hat{P}_{\text{ratio}}$ are min-max normalized values of the capped latency variance and pause ratio respectively, $C_{\text{success}}$ is the compile success rate, and $\alpha_1$--$\alpha_5$ are empirically determined weighting coefficients. The resulting score is clipped to the range $[0, 100]$ and mapped to a categorical \texttt{risk\_level} via fixed thresholds as specified in Table~\ref{tab:features}.

\section[Classification and Dataset Balancing]{Supervised Classification and Dataset Balancing Framework}
\label{sec:ml_framework}

\subsection{Binary Classification Task Definition}
The machine learning objective was defined as a binary classification problem: predicting whether a given 30-second behavioral window represents a \textbf{High Stress} state (post-exam Likert survey score $\geq 4$) or a \textbf{Low/Moderate Stress} state (Likert survey score $< 4$). This threshold was selected because survey responses of 4 or 5 on a 5-point scale correspond to clinically significant levels of perceived stress, and because the 4/5 group constituted approximately 83\% of all survey responses, representing the natural dominant class in the examination setting.

\subsection{Class Imbalance Resolution via SMOTE}
Because the raw training dataset contained a severe natural class imbalance (~83\% High Stress observations), any classifier trained on the unmodified dataset would suffer from majority-class bias, trivially achieving $\sim$83\% accuracy by always predicting ``High Stress.'' To overcome this imbalance trap, the Synthetic Minority Over-sampling Technique (SMOTE) was applied within each training fold.

\noindent SMOTE operates by evaluating the $k$-nearest neighbors within the minority class feature space (Low Stress samples $x_i$) and interpolating new synthetic data points along the vector connecting each sample to a randomly selected neighbor $x_j$:
\begin{equation}
x_{\text{new}} = x_i + \lambda\,(x_j - x_i), \quad \lambda \sim \mathcal{U}(0,\,1)
\end{equation}
This interpolation populates the sparse low-stress regions of the feature space without simply duplicating existing samples, which would cause overfitting. After applying SMOTE, each training fold achieved a perfectly balanced 50/50 class ratio, reducing the trivial random-guessing baseline from 83\% to exactly 50\%.

\subsection{10$\times$10 Repeated Stratified K-Fold Cross-Validation}
To obtain statistically robust and variance-controlled performance estimates, a 10$\times$10 Repeated Stratified K-Fold Cross-Validation protocol was adopted. In this setup:
\begin{itemize}
    \item The dataset was split into 10 stratified folds, preserving the original class distribution within each fold.
    \item SMOTE was applied exclusively within the \textbf{training portion} of each fold, never on the held-out test split, to completely prevent synthetic data leakage from contaminating the evaluation.
    \item The entire 10-fold process was independently repeated 10 times with different random seeds, yielding a total of \textbf{100 independent training and evaluation cycles} per model.
    \item This protocol produces highly stable accuracy estimates with quantified standard deviation, confirming the generalizability of each model's performance across the full dataset.
\end{itemize}

\section{Research Methodology}
\label{sec:methodology}

The overall research methodology followed a structured five-phase pipeline as illustrated in Figure~\ref{fig:methodology0}. Each phase is causally dependent on the previous, ensuring complete traceability from raw OS event capture through to final validated machine learning evaluation results.

\begin{figure}[htbp]
    \centering
    \includegraphics[width=\textwidth]{Methodology Diagram.jpg}
    \caption{Five-Phase Research Methodology Pipeline for Passive Anxiety Detection in Academic Programming Examinations.}
    \label{fig:methodology0}
\end{figure}

\begin{enumerate}
    \item \textbf{Phase 1  System Development:} The core OS-level monitoring script was developed in Python using native Windows API calls. The \texttt{.bat} script wrapper was engineered to enable silent, single-click deployment on student workstations with no manual configuration steps required from the student.

    \item \textbf{Phase 2 Data Collection:} The monitoring system was deployed on student workstations across four separate graded Class Test sessions involving 67 first-year undergraduate computer science students (all aged below 25). Within 5--15 minutes of each exam's conclusion, students completed a digital post-exam survey reporting their perceived stress on a 1--5 Likert scale.

    \item \textbf{Phase 3  Preprocessing and EDA:} The 67 individual CSV log files were merged into a unified dataset. Warm-up rows (zero-keystroke initial intervals) were discarded. Extreme \texttt{latency\_variance\_ms} outliers were capped at 30,000 ms. Exploratory data analysis was then conducted to characterize the label distribution, examine feature-level behavioral profiles across anxiety bands, and analyze inter-feature correlations via a heatmap.

    \item \textbf{Phase 4  Model Training:} SMOTE balancing was applied within each fold of a 10$\times$10 Repeated Stratified K-Fold Cross-Validation loop. Eight classical supervised machine learning classifiers were trained and evaluated over 100 independent training cycles each.

    \item \textbf{Phase 5  Evaluation:} Pearson and Spearman correlation tests validated the rule-based \texttt{anxiety\_score} against ground-truth Likert labels at the per-student level. Model accuracy, standard deviation, and feature importance rankings were reported to determine the most effective and interpretable classifier.
\end{enumerate}

\section{Summary}
\label{sec:ch3_summary}

This chapter established the complete design specification of the passive anxiety detection system as it was implemented and studied. The architectural blueprint demonstrated how three parallel OS-level sensing threads a keyboard hook, a window focus tracker, and a compile event detector converge every 30 seconds into a behavioral aggregator that produces an 18-feature privacy-preserving CSV log. The complete feature specification table and accompanying mathematical formulations (WPM, Latency Variance, Backspace Rate, Focus Switch Index, and the rule-based Anxiety Score) provide full scientific transparency over the system's input space and computational model. The SMOTE interpolation formula and 10$\times$10 Repeated Stratified K-Fold Cross-Validation protocol established the rigorous, leakage-free framework under which all machine learning evaluations were conducted. Finally, the structured five-phase research methodology, 20-week Gantt schedule, and individual task allocation matrix confirm the disciplined, reproducible execution of this study from system development through to final evaluation.
