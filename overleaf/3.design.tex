\chapter{Project Design}
\label{ch:project_design}
This chapter details the complete architectural blueprint, requirement analysis, behavioral feature engineering specifications, and project execution plan of the proposed passive anxiety detection system. The design is grounded entirely in the actual system built and deployed during the study: a zero-intrusion OS-level monitoring tool for first-year undergraduate students using the Code::Blocks C/C++ IDE during academic examination sessions.

\section{Requirement Analysis}
\label{sec:requirements}

\subsection{Functional Requirements}
The following functional requirements define the behavioral boundaries of the implemented system:

\begin{itemize}
    \item The system shall run a non-intrusive background monitoring script that passively captures keystroke timing metrics and active window context switches without interfering with the student's primary Code::Blocks environment.
    \item The system shall detect and log the state of the compiler executable (\texttt{gcc.exe}) via the Windows \texttt{tasklist} command to track compilation attempts during each 30-second interval.
    \item The system shall aggregate 18 behavioral features into a structured row and compute a rule-based \texttt{anxiety\_score} (range: 0--100) and categorical \texttt{risk\_level} (LOW, MODERATE, HIGH, CRITICAL) at the end of every 30-second window.
    \item The system shall serialize all computed feature rows into a local CSV log file, ensuring no literal keystrokes, alphanumeric input, or source code content is recorded.
    \item The system shall be launchable via a batch (\texttt{.bat}) script so that the researcher can start the monitor on each student workstation silently at the beginning of an exam session.
    \item The system shall support post-session offline analysis by producing a structured, machine-learning-ready dataset for statistical validation and supervised classification experiments.
\end{itemize}

\subsection{Non-Functional Requirements}
\begin{itemize}
    \item \textbf{Performance:} The background monitoring process must operate asynchronously at the OS level, consuming negligible CPU time. The 30-second aggregation window must not introduce any perceptible latency into the Code::Blocks compilation environment.
    \item \textbf{Privacy:} All telemetry must be strictly behavioral and structural. The system must never record literal characters, variable names, code contents, or passwords. Only timing intervals, counts, and window title hashes are permitted.
    \item \textbf{Scalability:} The script must be deployable across multiple student workstations simultaneously in a university computer lab without any networked infrastructure, cloud services, or IDE plugin installation.
    \item \textbf{Reproducibility:} The resulting CSV dataset must be structured such that any researcher can independently re-run the full data cleaning, EDA, and ML pipeline to reproduce the reported results.
    \item \textbf{Portability:} The \texttt{.bat} launcher and the underlying Python script must require only standard Python libraries and Windows API calls, avoiding dependency on commercial software.
\end{itemize}

\section{System Architecture}
\label{sec:system_architecture}

\subsection{Overview}
The system is architecturally decoupled into two distinct execution phases: the \textbf{Live Data Collection Phase} (runs on student machines during exams) and the \textbf{Offline Research Analysis Phase} (runs by the researcher after exams). This separation ensures zero interference with the student's workspace and allows the full ML pipeline to be executed at any time after the data has been collected.

\subsection{Architectural Layer Breakdown}

The live collection pipeline is structured into three parallel sensing threads that converge into a single aggregation engine:

\begin{enumerate}
    \item \textbf{Keyboard Hook Thread (100 Hz):} This thread attaches a low-level keyboard listener to the Windows OS event queue using the \texttt{WinAPI} hook mechanism. It records the precise timestamp of every key-press and key-release event without capturing the actual character typed. From these timestamps, it continuously computes the inter-keystroke interval (IKI), words-per-minute (WPM) rate, and backspace event count.

    \item \textbf{Window Tracker:} This component polls the \texttt{GetForegroundWindow()} API at regular intervals to determine which application currently holds the user's focus. It classifies the active window into one of three categories: the Code::Blocks editor (\texttt{codeblocks.exe}), a terminal/runtime window (\texttt{cmd.exe}), or an external application (e.g., a PDF reader). Each transition between categories is counted as a \texttt{focus\_switch} event.

    \item \textbf{Compile Detector:} This component runs the Windows \texttt{tasklist} command at regular intervals to check whether \texttt{gcc.exe} is actively executing. It tracks both the frequency of compilation attempts and their outcomes (success or error), contributing directly to the \texttt{compile\_success\_rate} feature.
\end{enumerate}

At the end of every 30-second tumbling window, the \textbf{30-Second Aggregator Engine} collects the outputs from all three threads and computes the full 18-feature behavioral vector. The rule-based anxiety scoring engine then maps this vector to a scalar \texttt{anxiety\_score} and a categorical \texttt{risk\_level}. The completed row is appended to the student's local CSV log file. The Code::Blocks IDE itself is completely unaffected throughout this entire process.

\section{Research Methodology}
\label{sec:methodology}

The overall research methodology followed a five-phase pipeline as illustrated in Figure~\ref{fig:methodology}. Each phase builds directly on the previous, ensuring full traceability from raw OS events to validated machine learning results.

\begin{figure}[htbp]
    \centering
    \includegraphics[width=\textwidth]{Methodology.PNG}
    \caption{Five-Phase Research Methodology for Passive Anxiety Detection in Academic Programming Examinations.}
    \label{fig:methodology}
\end{figure}

\begin{enumerate}
    \item \textbf{Phase 1 -- System Development:} The core monitoring script was developed using Python and the Windows API. The \texttt{.bat} launcher was created to allow silent one-click deployment on student machines.
    \item \textbf{Phase 2 -- Data Collection:} The script was deployed across 67 first-year undergraduate student workstations during four separate graded Class Tests. Post-exam Likert surveys (1--5 stress scale) were collected within 5--15 minutes of exam completion.
    \item \textbf{Phase 3 -- Preprocessing \& EDA:} Individual student CSVs were merged, warm-up rows were filtered, extreme \texttt{latency\_variance\_ms} values were capped at 30,000 ms, and exploratory data analysis was conducted to understand the label distribution and feature correlations.
    \item \textbf{Phase 4 -- Model Training:} SMOTE was applied within a 10$\times$10 Repeated Stratified K-Fold Cross-Validation loop to balance the 83\% class imbalance and evaluate 8 classification models without data leakage.
    \item \textbf{Phase 5 -- Evaluation:} Statistical correlation tests (Pearson and Spearman) validated the rule-based anxiety score against ground-truth survey labels. Final ML model performance was compared via mean accuracy across 100 training runs.
\end{enumerate}
