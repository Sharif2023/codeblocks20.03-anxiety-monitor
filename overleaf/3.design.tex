\chapter{Project Design}
This chapter highlights the requirement analysis, architectural abstractions, and structural interface design specifications of the system. It presents the functional dependencies, data mapping boundaries via context and data flow diagrams, and details the comprehensive multi-tiered layout architecture of the high-fidelity interactive UI/UX adaptive intervention prototype.

\section{Requirement Analysis}
\subsection{Functional and Nonfunctional Requirements}
\subsubsection{Functional Requirements}
\begin{itemize}
    \item The system must execute a non-intrusive background monitoring script capable of passively collecting user typing indicators and window context switches without interfering with primary editor performance.
    \item The system backend must feature a machine learning classification framework capable of mapping processed temporal and frequency features into distinct anxiety risk states in real time.
    \item The application infrastructure must define a distinct structural pipeline to log anxiety-related metrics into a local serialized file format for post-session analysis and validation.
    \item The system must implement robust data boundaries to protect user privacy, processing raw keystroke telemetry locally and excluding literal text or alphanumeric inputs from database compilation.
    \item The front-end intervention system must present a comprehensive, multi-tiered visual feedback loop implemented as an interactive UI/UX prototype layout designed specifically for desktop IDE environments.
\end{itemize}

\subsubsection{Nonfunctional Requirements}
\begin{itemize}
    \item \textbf{Performance:} The passive data extraction workflow must run asynchronously at the operating system level, maintaining zero-latency overhead within a 30-second tumbling batch window to avoid CPU throttling inside the C/C++ compiler context.
    \item \textbf{Usability:} The visual interface layouts must optimize developer focus, utilizing clean typography and non-intrusive spacing rules to completely eliminate notification fatigue during cognitively demanding tasks.
    \item \textbf{Scalability:} The architectural framework must remain decoupled, ensuring the underlying feature extraction script and machine learning parameters can scale fluidly across diverse user cohorts and multiple classroom evaluation sessions simultaneously.
    \item \textbf{Security:} All gathered telemetry matrices must remain anonymous, relying on local session hashes rather than real-world developer identifications to ensure ethical alignment with data safety standards.
\end{itemize}

\subsection{Data Flow Diagram}
The structural approach to real-time developer stress monitoring aligns closely with empirical paradigms established in human-computer interaction, pedagogical analytics, and machine learning classification. Prior research confirms that programming motivation increases significantly when supported by adaptive, emotionally intelligent tooling \cite{stemedu2025, programmingAnxiety2025}. Fine-grained mechanical indicators, such as keystroke dynamics and temporal variance, represent highly reliable tracking metrics for inferring underlying affective states \cite{keystroke2023, realtimeStress2012}, while combining continuous passive sensing with self-reported validation metrics provides the highest empirical accuracy \cite{mdpi5739}. The design outlined in this report bridges an essential gap in current software engineering literature: establishing an architectural bridge that links live behavioral telemetry models directly to custom mitigation interfaces mapped via high-fidelity prototyping.

As illustrated in Figure~\ref{fig:dfd}, the technical pipeline is broken down into modular operational steps. These inputs are structured into micro-window batches and passed directly to Module 2.0 (Detect Anxiety / Machine Learning Engine), which utilizes a pre-trained Support Vector Machine (SVM) model to classify stress tiers. The classified risk state updates the system activity database logs via Module 4.0, while concurrently prompting Module 3.0 (Trigger Context-Aware Intervention) to display localized adaptive feedback templates optimized for the detected level of cognitive distress.

\section{Detailed Methodology and Design}
\label{sec:detailed_methodology_and_design}

The systemic architecture of the real-time anxiety tracking engine and the conceptual framework driving the high-fidelity UI/UX adaptive intervention matrices require a decoupled, low-overhead software engineering execution. To ensure that student tracking remains non-invasive, lightweight, and completely transparent, the system design avoids heavy IDE hooks that create version mismatches and resource contention. Instead, the architecture establishes a system-level pipeline that bridges live telemetry collection directly to an optimized machine learning validation loop.

\subsection{Architectural Layer Breakdown}
The operational pipeline is architected into three distinct structural layers that run in a continuous feedback loop at a 30-second temporal resolution:

\begin{enumerate}
    \item \textbf{The Data Sensing and Extraction Layer (OS API Bridge):} This layer runs asynchronously within the underlying operating system environment. Rather than injecting native plugins into specific editor compilation setups—which could introduce thread locking or degrade workspace performance—this engine targets the primary OS interaction channels. It intercepts high-frequency user interactions by implementing continuous low-level window loops and key event notifications. This allows the system to determine which binary processes currently occupy active foreground focus (e.g., distinguishing when focus shifts between \texttt{codeblocks.exe}, \texttt{gcc.exe}, or \texttt{cmd.exe}). Concurrently, the mechanical keyboard interaction loop records timing variations between successive key releases without caching literal character arrays, enforcing strict privacy boundaries.
    
    \item \textbf{The Feature Engineering and Processing Layer (Intelligence Pipeline):} Once raw tracking matrices are serialized over a 30-second tumbling window, this processing layer transforms unconditioned event counts into clean, multi-dimensional feature coordinates. The primary operations consist of data cleaning, warm-up filtering, and mathematical standardization. This engine isolates specific metrics such as typing speed words-per-minute (WPM), latency variance averages, and focus switch tracking values, formatting them into structured entries for classifier inference.
    
    \item \textbf{The UI/UX State Machine Adaptivity Layer (Mitigation Engine):} This layer forms the theoretical core behind the high-fidelity interactive prototype matrices. It maps out explicit visual mitigation pathways tailored to the specific output class computed by the supervised machine learning backend. When stress thresholds are breached, this state machine dynamically dictates feedback parameters across four structural modes (Low, Moderate, High, Extreme), matching interface behaviors to individual cognitive distress markers.
\end{enumerate}

\subsection{Mathematical Formulation of Behavioral Feature Telemetry}
To accurately model developer stress, the system translates noisy raw human-computer interaction logs into a high-frequency feature space. For any given 30-second observation interval $\Delta t$, let $K$ represent the total number of recorded key release events, and let $T_i$ represent the precise timestamp of the $i$-th key event. The mathematical equations governing the primary behavioral predictors are formulated as follows:

\subsubsection{1. Mean Typing Velocity ($V_{\text{WPM}}$)}
The standard calculation for typing velocity measures the rate of alphanumeric output normalized to a words-per-minute metric, assuming an average word layout length of five characters:
\begin{equation}
V_{\text{WPM}} = \frac{\left( \frac{K}{5} \right)}{\Delta t \times \frac{1}{60}} = \frac{12 \times K}{\Delta t}
\end{equation}

\subsubsection{2. Typing Latency Variance ($\sigma^2_{\text{Lat}}$)}
Let the individual latency interval $L_i$ define the precise duration between two successive key release timestamps, where $L_i = T_i - T_{i-1}$. The mean latency over the window interval is expressed as:
\begin{equation}
\mu_{\text{Lat}} = \frac{1}{K-1} \sum_{i=2}^{K} L_i
\end{equation}
To isolate erratic, non-rhythmic typing patterns—which serve as strong indicators of underlying cognitive hesitation or situational panic loops—the system calculates typing latency variance:
\begin{equation}
\sigma^2_{\text{Lat}} = \frac{1}{K-2} \sum_{i=2}^{K} (L_i - \mu_{\text{Lat}})^2
\end{equation}
During preprocessing, to eliminate mechanical outliers caused by external disruptions, the feature values are bounded by a strict upper capping threshold:
\begin{equation}
\sigma^2_{\text{Lat, capped}} = \min(\sigma^2_{\text{Lat}}, 30000\,\text{ms})
\end{equation}

\subsubsection{3. Deletion Percentile Index ($R_{\text{Backspace}}$)}
Let $B$ represent the explicit aggregate count of backspace or delete events executed during the window sequence. The overall backspace rate percentage is calculated as a relative fraction of global keystroke activities:
\begin{equation}
R_{\text{Backspace}} = \left( \frac{B}{K + B} \right) \times 100
\end{equation}

\subsubsection{4. Spatial Environment Fragmentation ($F_{\text{Switch}}$)}
Let $W_t$ represent the window descriptor text hash occupying the OS foreground focus at micro-timestamp $t$. The context focus-switching index tracks active environmental toggling, counting transitions where the current focus changes from the previous interval:
\begin{equation}
F_{\text{Switch}} = \sum \mathbb{I}(W_t \neq W_{t-\delta})
\end{equation}
where $\mathbb{I}$ functions as an indicator variable returning a value of 1 if the conditional argument is true, and 0 otherwise. A significant increase in $F_{\text{Switch}}$ values captures fragmented attention spans, providing a robust, non-intrusive signature of elevated mental workload.

\subsection{Supervised Classification and Dataset Balancing Framework}
Because the dataset was gathered during live university examinations, the target classes exhibited an inherent, high natural imbalance, with approximately 83\% of observations falling into the high stress zones due to real evaluation pressures. Traditional machine learning models trained on highly imbalanced data will reliably suffer from majority-class bias, achieving false high accuracy rates simply by predicting the dominant label exclusively.

To resolve this imbalance, the system pipeline incorporates a two-stage preprocessing protocol:
\begin{itemize}
    \item \textbf{Synthetic Minority Over-sampling Technique (SMOTE):} This method dynamically balances training folds by interpolating synthetic coordinates within the feature space of the minority class (\texttt{LOW} stress). Rather than relying on simple row replication—which leads to overfitting—SMOTE evaluates the $k$-nearest neighbors for a minority sample $x_i$, calculating new nodes along the vector segments:
    \begin{equation}
    x_{\text{new}} = x_i + \lambda (x_j - x_i)
    \end{equation}
    where $x_j$ is a randomly selected nearest neighbor, and $\lambda$ represents a uniform random variable bounded within $[0, 1]$. This populates the sparse feature zones, balancing the training classes to a clean 50/50 ratio.
    \item \textbf{Decoupled Cross-Validation Safeguards:} To protect evaluation integrity, SMOTE balancing is executed strictly within the internal training loops of a 10$\times$10 Repeated Stratified K-Fold Cross-Validation cycle. The validation data splits are kept completely free of synthetic parameters, ensuring that the model is tested exclusively against genuine, unconditioned student interaction profiles.
\end{itemize}

\section{Project Plan}
\label{sec:project_plan}

The structural execution of this research spans two distinct academic trimesters, mapping out a sequential path from early concept creation and data collection scripts to machine learning engine evaluation and interactive UI/UX prototyping. 

To maintain transparency, the operational roadmap was divided into six key development milestones:
\begin{itemize}
    \item \textbf{Milestone 1 (Weeks 1--4): Literature Survey and Tool Selection.} Conducting extensive analysis of existing developer mental health informatics tools, establishing project requirements, and choosing the target IDE workspace ecosystem.
    \item \textbf{Milestone 2 (Weeks 5--8): Telemetry Hook Script Development.} Constructing the low-level asynchronous background tracking loop to record non-literal keystroke dynamics and window context switches at a 30-second interval.
    \item \textbf{Milestone 3 (Weeks 9--12): Examination Field Data Collection.} Deploying the automated collector scripts across multiple laboratory workstations to aggregate over 11,000 distinct interaction intervals from a study cohort of 67 first-year computer science students during graded exams.
    \item \textbf{Milestone 4 (Weeks 13--16): Preprocessing and Dataset Balancing.} Designing the data cleaning pipeline to cap mechanical outliers and deploying SMOTE over-sampling algorithms to resolve natural class imbalances.
    \item \textbf{Milestone 5 (Weeks 17--20): Machine Learning Pipeline Evaluation.} Training, tuning, and evaluating multiple classification architectures (including SVM, Random Forests, and Naive Bayes) via 10$\times$10 Repeated Stratified Cross-Validation loops.
    \item \textbf{Milestone 6 (Weeks 21--24): High-Fidelity UI/UX Prototyping.} Wireframing and connecting the multi-tiered adaptive intervention user interface frameworks within Figma, establishing the design specification guidelines for the system.
\end{itemize}
